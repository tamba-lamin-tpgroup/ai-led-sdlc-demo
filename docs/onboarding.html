Developer Onboarding

Build Salone Explorer the AI-led way

This repository ships more than an app specification. It ships an AI-led software development lifecycle: a harness of rules, sessions, governance hooks, slash commands, skills, and subagents that lets Claude Code do roughly 80% of the engineering while a human keeps the accountable 20%. This page explains every moving part, how they interlock, and why the result is cheaper in tokens, faster to ship, and harder to get wrong.

Stack: Vite + React + TypeScript Supabase + Tailwind Deploy: Vercel Harness: .claude/ Source of truth: SPEC.md

What you are looking at

A one-paragraph mental model before the detail.

Salone Explorer is a Sierra Leone tour-guide single-page app published by TpGroup (SL) Limited under the FambulTik brand. The product is defined entirely in SPEC.md. The .claude/ directory is the factory that turns that spec into code: it tells Claude Code what the rules are, where it is in the work, what it is allowed to do, and how to verify what it built. Nothing here is decorative. Every file either constrains the AI, gives it memory, or gives it a repeatable procedure.

The one idea

Separate the intent (SPEC.md), the guardrails (rules + hooks), the memory (sessions), and the procedures (skills + commands + agents). Keep them distinct and the AI can move fast without drifting, because every step is traceable to a source and checked by a gate.

The repository and its keystone rule

You cannot understand the tooling without the architecture it protects.

Every file in src/ belongs to exactly one of three layers. This is the rule most likely to be violated by an AI agent and the one that causes the most damage, so almost half the harness exists to defend it.

LayerPathHolds
Codesrc/components/, src/pages/, src/lib/React components, routing, business logic. No strings. No facts.
Datasrc/data/*.jsonAttraction records, region lookups, taxonomies.
Contentsrc/content/*.jsonAll user-facing strings, page copy, the disclaimer.

Code reads facts through the attractions repository (never importing JSON directly) and reads copy through t("namespace.key"). A hard-coded English string or attraction name anywhere under src/components/, src/pages/, or src/lib/ is a defect a grep will catch.

Why it matters

The separation is the bridge to a future CMS (Phase 8, Payload). Code that reads facts through a repository and copy through t() swaps its backing store - file JSON today, Postgres via Supabase tomorrow, a CMS later - without a rewrite. Code that inlines facts and strings has to be rebuilt. The harness refuses to let convenience erode that boundary.

The AI-led harness and the 80 / 20 split

A responsibility model, not a capability model.

The operating model divides the lifecycle between the AI and the human. The split is not about what Claude can do - Claude can do the human's 20% too. It is about who is accountable. Routing the accountable steps through a human is exactly what makes this AI-led rather than AI-only.

The AI does the 80%

Read SPEC + issue. Produce the context pack. Write the plan. Implement code, data, content, tests, and docs inside the three-layer rule. Run the verification loop. Spawn first-pass code and security review. Update README, headers, ADRs. Open the draft PR.

The human does the 20%

Approve the plan before implementation. Sign off on architecture (ADRs) and any SPEC deviation. Review the draft PR and apply judgement. Resolve ambiguous requirements. Final security and business sign-off. Click merge - Claude is denied that action.

Hard boundaries (enforced, not requested)

Claude must not gh pr merge, git push --force, git reset --hard, or bypass hooks with --no-verify. It must not commit secrets, and it must stop and ask before deviating from the architecture, the Attraction type, the Supabase schema, the brand, accessibility, or the project structure. These are enforced by settings.json deny-rules and hooks - see below.

The tool catalog

Nine kinds of component. Each has one job.

The harness is layered: rules set the law, sessions hold memory, hooks enforce mechanically, commands and skills are the procedures you run, subagents are specialists you delegate to, and MCP servers give the AI fast, structured access to code, libraries, and GitHub. The remaining pieces - usage tracking and the SEO/GEO/AEO suite - are capabilities layered on top.

Sessions .claude/sessions/ · durable memory

A session file is the only memory the next session has. It records goals, timestamped updates, findings, decisions, blockers, outcomes, and the next-step handoff. Sessions are mandatory for development work.

/session-start <phase|issue#>   open a session, capture goals
/session-update                 append a structured update after each step
/session-current                show full state (goals, updates, blockers)
/session-resume <id>            reactivate a paused session with its context
/session-end                    write outcomes + next-session handoff, clear pointer
Why it matters (and saves tokens)

Without durable memory, every new session re-reads and re-derives the project from scratch - expensive in tokens and error-prone. The session file lets the next run start from a written summary of decisions and findings instead of rebuilding context. The SessionStart hook surfaces it automatically.

Governance hooks .claude/hooks/ · wired in settings.json

Hooks are the teeth. They run automatically on Claude Code lifecycle events and can block an action before it happens. The rules describe the law; the hooks enforce it mechanically so compliance does not depend on the model remembering.

EventHookEnforces
SessionStartsession-start.shLoads active-session memory and recent history into context.
UserPromptSubmituser-prompt-submit.shBlocks code-producing prompts that cite no SPEC phase/section or issue.
PreToolUse(Bash)pre-tool-bash.shSecret-scans, requires a commit trailer, blocks force-push, reset --hard, --no-verify, gh pr merge.
PreToolUse(Edit)pre-tool-edit.shBlocks new source files missing a one-line purpose header.
PostToolUse(Edit)post-tool-edit.shAuto-runs ESLint --fix + Prettier on the edited file.
Stopstop-session.shNags if a session is left open; snapshots the transcript.

Backing these, settings.json deny-rules hard-block the dangerous commands at the permission layer too (defense in depth): git push --force, git reset --hard, git branch -D, git clean -f, commit --no-verify, gh pr merge, rm -rf /, rm -rf ~, and reads of .env, .pem, .key, and id_rsa files.

Subagents .claude/agents/ + global · specialists you delegate to

A subagent runs in its own context window and returns only its conclusion. That isolation is a token-saving and focus device: a fan-out search reads many files but hands back a summary, not the file dumps, keeping the main thread lean.

security-reviewersonnetDiffs touching auth, secrets, Supabase/RLS, env vars, rendering. Blocks handoff on high findings.

Three more are installed globally and shared across all projects: graph-explore (graph-first codebase navigation), graph-impact (blast-radius before a refactor), and graph-review (risk-scored review). They are backed by the code-review-graph MCP and report using roughly 70-90% fewer tokens than text-based search for the same questions.

MCP servers .mcp.json

code-review-graph

A queryable knowledge graph of the codebase. Structural questions ("what depends on X", "where is Y defined") resolve by graph lookup instead of repeated full-text reads - faster and far cheaper in tokens.

context7

Live, version-correct library documentation on demand. Stops the AI guessing an API from stale training data.

Issues, PRs, and project boards. Lets /orchestrate read an issue and the workflow open a draft PR.

filesystem

Scoped file access for the harness tooling.

Usage tracking statusline + /tokens + session snapshot

You cannot optimize what you do not measure. Three mechanisms make token and time cost visible:

  • Live statusline - statusline.sh shows the branch, issue, session, diff, and a live time:<elapsed> cost:$<usd> read straight from Claude Code's own status payload.
  • /tokens - runs token-report.py over this project's transcripts only, deduped by message id, Claude-only. Exact input/output/cache token counts per day or per session. No cross-project or cross-tool leakage.
  • Session-end snapshot - /session-end stamps the session file with total coding time and a token summary, so each session leaves a durable record.

Cost in the report is opt-in: rates live in .claude/config/model-pricing.json and ship as null, so no price is ever fabricated. The live statusline already shows Claude Code's authoritative cost.

Each step is wrapped by an enforcement layer. You do not have to remember the rules - the harness refuses the action if you stray.

The wiring in one sentence

SPEC.md sets intent -> the orchestrator proves the work is traceable and routes it -> plan-then-code gets human approval -> rules and hooks constrain every edit and command -> the verification loop and reviewers prove quality -> sessions record what happened -> a human merges.

Why this exists - the payoff

Four returns: cheaper tokens, faster flow, higher quality, and security/performance by construction.

1. It saves tokens

  • Structured retrieval over brute-force reading. The code-review-graph MCP and the graph subagents answer structural questions by graph lookup - their own definition cites roughly 70-90% fewer tokens than text search for the same task.
  • Subagents return conclusions, not file dumps. A fan-out search burns its tokens in an isolated context and hands back a summary, so the main thread never pays to hold every file it touched.
  • Durable session memory. The next session reads a written summary of decisions and findings instead of re-deriving the whole project - the single biggest repeat-cost in long-running AI work.
  • context7 instead of guess-and-correct. Pulling the correct library API once is cheaper than generating wrong code, failing, and regenerating.
  • The three-layer rule prevents rework. Code that would have to be rebuilt to reach the CMS is the most expensive kind of token spend - the rework. The rule eliminates it up front.
  • You measure it. /tokens and the live statusline turn token cost from invisible to a number you watch, so waste gets caught.

2. It makes the workflow efficient

  • One pipeline, not a blank page. Every task follows the same loop, so there is no per-task improvisation about process.
  • Auto-formatting on save removes lint/format churn from the human loop entirely.
  • The orchestrator pre-flight catches missing specs, unmet phase preconditions, and unmet dependencies before an engineer starts - failing fast instead of mid-build.
  • Specialist routing. Vague issue -> product-analyst; cross-layer -> architect; data/content -> the SME gate. The right method, automatically.

3. It ensures high quality

  • Quality is a gate, not a hope. The verification loop must be green before handoff: lint, typecheck, unit, a11y, build, and the three-layer grep.
  • Two independent reviewers (code and security) run on every diff, tuned to the failure modes AI-generated code is statistically prone to - silent fallbacks, dead error handling, spurious abstractions, missing tests.
  • An SME content gate blocks invented attraction facts; unconfirmed facts must be marked, never guessed.
  • Accessibility is CI-blocking - serious or critical axe violations on the five smoke routes fail the build.

4. It is secure and performant by construction

  • Secrets cannot be committed - the scanner blocks the commit, and reads of .env/.pem/.key are denied at the permission layer.
  • Only VITE_-prefixed env vars reach the browser; the Supabase service-role key never does.
  • Row-Level Security on every user-scoped table with auth.uid() = user_id policies - the security-reviewer blocks handoff on RLS gaps.
  • Performance and security are first-class non-functional requirements in the engineering rules - designed in up front, not bolted on. The SEO/GEO/AEO suite and the a11y gate keep the shipped site fast and discoverable.

AI-led SDLC best practices, and why they matter

The principles the harness encodes. Each exists because skipping it has a known cost.

Spec-first, always

No code without a traceable source. Why: an AI will happily build the wrong thing convincingly. Tracing every change to a SPEC section or issue keeps output aligned with intent and makes review tractable.

Plan before code

Produce a plan and get human approval before editing production files. Why: the cheapest place to catch a wrong approach is the plan, before any tokens or trust are spent on an implementation.

Keep humans on the accountable steps

Plan approval, architecture sign-off, and merge stay human. Why: AI-led is a responsibility model. Accountability for irreversible decisions must sit with a person.

Separate intent, memory, and procedure

SPEC vs sessions vs skills. Why: conflating them is how context windows overflow and drift creeps in. Clean separation keeps each artefact small and selectively readable.

Enforce mechanically, not by reminder

Hooks block; they do not advise. Why: a model that "knows" a rule still breaks it under pressure. A hook that blocks the commit does not.

Fail fast, no silent fallbacks

Validate at every boundary; raise loudly with context. Why: a silent fallback hides the bug until production. Explicit failure with the failing input is debuggable.

Smallest viable diff

Edit only what the phase requires; no drive-by refactors. Why: small diffs review faster, regress less, and keep token cost proportional to the actual change.

Verify before handoff

The full gate runs green before a PR opens. Why: review time is the scarce resource. Never spend a human's attention on something a machine could have caught.

Write the memory down

End every session with outcomes and a next-step handoff. Why: undocumented context is lost context, and lost context is re-derived at full token price.

Measure cost continuously

Token and time usage are always on screen. Why: efficiency you cannot see is efficiency you cannot improve.

Getting started

One-time activation, then the daily loop.

One-time activation (human only)

Two files configure Claude's own permissions and cannot be auto-written by Claude. Activate them yourself, review first, then restart Claude Code.

# 1. Install the harness settings (already done in this repo if settings.json exists).
mv .claude/settings.json.proposed .claude/settings.json

# 2. Create your local, gitignored overrides + GitHub token for the MCP.
cp .claude/settings.local.json.template .claude/settings.local.json
#    then set GITHUB_PERSONAL_ACCESS_TOKEN in settings.local.json

# 3. Restart Claude Code so it loads settings.json, hooks, and .mcp.json.

After this the status line, hooks, lead-developer output style, and MCP servers are live. Node 20+ is required for the app commands once Phase 1 has scaffolded package.json.

Your first task, end to end

/session-start phase-3-accessibility    # or /session-start <issue#>
/orchestrate <github-issue-url>         # pre-flight + context pack + routing
# ... the plan is presented; a human approves it ...
# ... implement inside the three-layer rule, /session-update as you go ...
/quality                                # full gate: lint, types, tests, a11y, secrets, build
/code-push "feat: ..."                  # conventional commit + trailer + push to dev
/handoff                                # verification + reviews + draft PR
/session-end                            # outcomes + next-session handoff
/tokens                                 # see what the session cost

For long, low-supervision runs: /session-autonomous-coding <issue#>.
