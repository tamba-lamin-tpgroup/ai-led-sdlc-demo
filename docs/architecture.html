Architecture Reference

How Salone Explorer is built

Salone Explorer is a Sierra Leone tour-guide single-page application published by TpGroup (SL) Limited under the FambulTik brand. This document describes its architecture across six views - business, development, platform, application, infrastructure, and runtime - plus the AI-led SDLC harness that produces it, the testing approach, and the security posture. Every claim traces back to SPEC.md, the single source of truth.

Vite + React + TypeScript Supabase + Tailwind Deploy: Vercel Harness: .claude/ WCAG 2.2 AA Source of truth: SPEC.md

The shape of the system

Source SPEC §1, §2, §5, §17

Salone Explorer ships in two halves. Phase 1 is a fully static, public-facing site - a curated catalogue of Sierra Leone attractions rendered from local JSON, SEO/AEO/GEO optimised, structured-data-rich, and deployed to Vercel. Phase 2 adds a Supabase backend for authentication, bookmarks, favorites, account view, and tour scheduling. Three post-class phases (9-11) extend it toward a transactional product: real-time reviews, transactional email, and Stripe deposits. A future Phase 8 swaps the file-backed content for Payload CMS.

The architecture is organised around one keystone idea: a strict separation of code, data, and content, with all data access funnelled through a repository abstraction. That single decision is what lets the backing store move from JSON to Postgres to a CMS without rewriting the UI. Everything else in this document hangs off it.

Disclaimer

Salone Explorer is built for demonstration and educational purposes only. It is not an official tourism service. Attraction details are curated from public sources and may be out of date. No payments or real bookings are processed in the in-class build (Phases 1-7). Enabling Stripe (Phase 11) requires revising the §17 disclaimer. Rendered verbatim in the footer, on /about, and in the Home alert.

Sourcing and legal posture

All attraction facts derive from five authoritative sources (the National Tourist Board, Sierra Leone Tourism, Visit Sierra Leone, Sierra Leone Heritage, and sierra-leone.org). Facts are paraphrased, never copied; every record cites at least one source URL; unconfirmed hours fall back to "Hours vary - confirm locally". 2. Development architecture

The keystone: three layers, one repository abstraction, and the structure that keeps them apart.

Source SPEC §5, §6.1, §12 · rules three-layer-separation, api-conventions

The three layers

Every file belongs to exactly one layer. The layer dictates where it lives. No component, page, or library file may contain a user-facing string, an attraction fact, or a data fixture.

Code, data, and content are physically separated; only the code layer reads the other two, and only through indirection.

src/components/ · src/pages/ · src/lib/ - React components, routes, business logic. No strings. No facts.

attractions.jsonregions.jsontypes.ts
Content layer
src/content/*.json - all UI strings, page copy, microcopy, the disclaimer. Eventual home: Payload globals (P8).
strings.en.jsonflat key -> string
The hard rule

If you are about to type "Schedule a Tour" or "Tiwai Island" inside a .tsx file - stop. The string goes in strings.en.json and is read via t("namespace.key"). The fact goes in attractions.json and is read through the attractions repository. A grep for English strings or attraction names under src/components, src/pages, or src/lib (excluding src/lib/content) must return zero matches.

The repository pattern

Components consume attractions from the barrel src/lib/content/index.ts. The barrel selects an implementation from VITE_ATTRACTIONS_SOURCE. The UI never knows which backend serves the data - this is the entire point of the architecture and the bridge to Phase 8.

One interface, swappable implementations, chosen by environment variable.

Strings indirection

src/lib/content/strings.ts exports t(key). All copy flows through it. Strings never contain HTML - rich text is composed in components from multiple keys. Adding a new key is a documented PR step. A second locale file (strings.kri.json for Krio) is the i18n on-ramp.

The shippable app and the AI-led SDLC tooling live in separate trees. Branching and commits

Two long-lived branches: dev (integration trunk) and main (production, deployed to Vercel). Promotion is one-directional, dev -> main, gated on four green CI workflows plus one human approval. Feature branches are issue-<num>-<slug>, cut from dev. Every commit carries a traceability trailer (Requirement: / Phase: / Incident:), enforced by a hook.

Deploy boundary

The app lives in the salone-explorer/ subdirectory and Vercel's Root Directory is set to it. Vercel clones the whole repo but builds only from that folder, so the .claude/ harness, docs/, and the spec are excluded from the deployment by construction. CI workflows live at the repo root and run with working-directory: salone-explorer. See ADR-09.

The four gates

WorkflowChecks
ci.ymlnpm ci, ESLint (incl. jsx-a11y), tsc --noEmit, npm run build.
codeql.ymlGitHub CodeQL, security-and-quality queries.
security.ymlSemgrep (owasp-top-ten, js/ts/react, secrets) + npm audit --audit-level=high.
a11y.yml@axe-core/playwright on /, /attractions/tiwai-island, /about, /signin, /signup; fails on serious/critical.

Backend provisioning and compute

Supabase is provisioned (Phase 5) via the Vercel integration or manually, then the §6.3 schema is applied with RLS before any auth feature is built. Server-side compute for the post-class phases runs in Supabase Edge Functions (Deno) - they hold Stripe and email secrets, handle webhooks, and send mail, so no secret reaches the browser.

The secret boundary

Only VITE_-prefixed variables are exposed to the browser: VITE_SITE_URL, VITE_ATTRACTIONS_SOURCE, VITE_SUPABASE_URL, VITE_SUPABASE_ANON_KEY, VITE_STRIPE_PUBLISHABLE_KEY. Server secrets (STRIPE_SECRET_KEY, STRIPE_WEBHOOK_SECRET, EMAIL_PROVIDER_API_KEY) live only as Edge Function secrets. OAuth client secrets live in the Supabase dashboard, never in the client (§16).

The detail page subscribes to Supabase Realtime over the reviews table. Published reviews stream in live; an auth-gated form submits one review per user per attraction. Only status = 'published' rows are publicly visible (RLS), and the published set feeds the aggregateRating in the JSON-LD graph - closing the loop with AEO (§9.3, §13.3, §14).

8. Testing approach

Layered gates, cheapest first, with accessibility as a hard CI condition.

Source SPEC §10, §18, §20 · rule test-conventions · skills verification-loop, tdd-workflow

The test pyramid for Salone Explorer - and where each runs.

Unit (Vitest)
Lowest-cost behaviour gate. Pure functions and small units - repository shaping, t() resolution, JSON-LD builders. One test per branch. Behaviour-named. No network.
Accessibility (Playwright + axe)
@axe-core/playwright against WCAG 2.2 AA across five smoke routes. Mirrors the a11y.yml gate; fails on serious or critical violations.
E2E (Playwright)
Reserved for the highest-visibility user journeys. Slow and flaky-prone; needs a running server. Used sparingly.

Verification loop

The single quality gate before /handoff and at any natural pause: secret scan, lint, typecheck, unit tests, build, a11y smoke, and the three-layer grep checks. It returns a pass/fail report; work does not proceed past a failure. The quality command wraps it with an optional E2E step and prints a PASS/FAIL matrix.

TDD where it fits

For well-specified, behaviour-driven changes, the tdd-workflow authors a failing Vitest test per acceptance criterion, confirms it fails for the right reason, writes the minimum code to pass, then refactors. Acceptance criteria live in SPEC §20, per phase.

Accessibility posture

WCAG 2.2 AA is a hard requirement, not a nice-to-have. Automated: axe in CI plus Lighthouse a11y ≥ 95. Manual: a keyboard-only walkthrough and one screen-reader pass per release. Linting: eslint-plugin-jsx-a11y (recommended); errors fail the build and may never be disabled to make a build pass (§10).

Non-negotiable

Never disable a feature, assertion, lint rule, or check solely to make a test or build pass. Fix the root cause. The three-layer grep (zero hard-coded strings or attraction facts under src/components, src/pages, src/lib) is part of the gate.

Architecture decisions

The cross-cutting decisions, each with its rationale and its SPEC anchor. Formal ADRs live in docs/adr/; these are the standing decisions the SPEC already encodes.

ADR-01

Strict three-layer separation (code / data / content)

Decision: No component, page, or library holds a user-facing string or an attraction fact. Strings live in src/content/, facts in src/data/.
Why: Removes the editorial bottleneck, makes i18n possible, and prevents content drift. It is the bridge to Phase 8 (Payload CMS).
Source: SPEC §5, §5.5 · rule three-layer-separation
ADR-02

Repository pattern over the data layer

Decision: All data access goes through an AttractionRepository interface selected by VITE_ATTRACTIONS_SOURCE. Components never import JSON directly.
Why: Swap JSON -> Supabase -> Payload without touching UI code. Proven by the Phase 2.5 migration that changes no component.
Source: SPEC §5.2.3, §5.3 · rule api-conventions
ADR-03

File-first store, database optional

Decision: Phase 1 reads attractions from JSON. Supabase is the optional Phase 2.5 migration, not a Phase 1 dependency.
Why: Ships a deployable site in ~45 minutes with zero backend; defers infrastructure cost until auth genuinely needs it.
Source: SPEC §2, §5.2
ADR-04

Supabase with RLS on every user-scoped table

Decision: User data lives in Postgres with row-level security and auth.uid() = user_id policies, applied in the same migration as the table.
Why: Authorisation enforced at the database, not the client. No table ships without its policy.
Source: SPEC §6.3, §6.6, §15
ADR-05

Supabase Edge Functions for server compute

Decision: Stripe and email run server-side in Edge Functions (Deno), holding all secrets and verifying webhooks.
Why: No secret reaches the browser bundle; privileged writes (payment status) happen only server-side via the service role.
Source: SPEC §3, §15, §16
ADR-06

Prerendering for crawlers and LLMs

Decision: vite-plugin-prerender emits static HTML for /, /about, and every attraction; pages carry a JSON-LD @graph and llms.txt.
Why: SEO, GEO, and AEO all require machine-legible, scroll-free answers in static HTML.
Source: SPEC §3, §13, §14
ADR-07

Stripe-hosted checkout for PCI scope

Decision: Payments use Stripe Checkout/Elements; the app never handles card data. Status is confirmed only by a signature-verified webhook.
Why: Keeps PCI scope with Stripe and prevents client-forged payment state.
Source: SPEC §3, §6.6, §9.4, §15
ADR-08

Payload CMS deferred to Phase 8

Decision: Editorial content stays file-based until a future Phase 8 introduces a payloadAttractionRepository; Supabase keeps user-generated data, Payload owns editorial.
Why: The repository pattern built on day one is the recognised bridge; no rewrite needed when the team is ready.
Source: SPEC §5.4, §21
ADR-09

App isolated in a subdirectory, separate from the harness

Decision: The Vite app is scaffolded into salone-explorer/; the .claude/ harness and docs/ stay at the repo root. Vercel's Root Directory is set to salone-explorer; CI runs with working-directory: salone-explorer.
Why: The production deployment never contains the AI tooling or docs - the boundary is explicit and enforced by construction, not by an ignore list. The app stays self-contained and portable.
Source: SPEC §12, §18, §19 · ADR 0001

Principles

The binding engineering principles every agent and human follows. Each maps to a rule or a SPEC section.

Source rule engineering-principles · CLAUDE.md · SPEC §22

Architecture over quick fixes

Choose the right shape, not the fast one. Plan before code; justify the decision first. Quick fixes that defer cost are technical debt.

Fail fast, no silent fallbacks

Errors raise loudly with context. Validate explicitly at every boundary - route params, form input, env var, Supabase response, repository return.

Security up-front

Think OWASP top-ten and stack-specific risks (XSS, open redirect, missing auth, RLS gaps, exposed VITE_ secrets) before writing, not after.

No magic values

Constants in named config or design tokens; environment-specific values via VITE_ vars. No hardcoded defaults.

Stack-idiomatic style

TS strict, no any, function components + hooks, repository pattern for data, RLS on every user table, tokens for every visual.

Smallest viable diff

Edit only what the phase or issue requires. No drive-by refactors. Granular, testable TODOs.

SPEC is the source of truth

No code without a traceable source - a SPEC phase/section, a tracked issue, or an incident. SPEC wins over CLAUDE.md and the rules when they disagree.

Stop and ask

Do not deviate from SPEC §5 (architecture), §6.1 (type), §6.3 (schema), §8 (brand), §10 (a11y), or §12 (structure) without asking. When ambiguous, ask - do not guess.
